# 產品需求文件 (PRD)：戀愛互動式故事網站

**專案名稱**：戀愛互動式故事網站
**組別**：等一下要吃什麼? / 17
**日期**：2026-05-14

---

## 1. 產品概述 (Product Overview)

### 1.1 產品背景與動機
現今使用者在手機上閱讀故事或小說時，多為被動接收內容，容易產生審美疲勞。年輕一代的讀者不再滿足於旁觀者的角色，而是渴望在故事中尋求自我投射與互動體驗。本專案旨在解決讀者「對既定結局感到遺憾卻無力改變」的痛點，建構一個由讀者驅動的互動平台。結合生成式 AI 技術，讓使用者的每一次抉擇都能即時觸發獨特的劇情分支，將傳統的單向閱讀體驗轉化為雙向的情感互動。

### 1.2 目標受眾 (Target Audience)
- 對故事品質有要求的使用者。
- 喜愛分享互動體驗的年輕族群。
- 希望在虛擬世界中尋求互動與自我投射的讀者。

### 1.3 產品目標 (Product Goals)
1. **增加故事深度**：擴展故事線路，提供多元的分支結局。
2. **提升參與感**：透過客製化角色或互動選擇，強化讀者的情緒連結。
3. **完善進度管理**：建立會員登入系統，確保讀者能跨裝置同步閱讀進度。
4. **靈活存取**：提供「任意存檔點」功能，讓讀者能隨時回溯或保存特定劇情點。
5. **沉浸式體驗**：提供符合劇情的背景音樂及精美的動畫與轉場。

## 2. 需求範圍與限制 (Scope & Constraints)

### 2.1 不包含範圍 (Out of Scope)
- **無高成本配音**：本次專題不包含專業的語音配音。
- **無高成本 3D 建模**：以 2D 圖片與文字互動為主。

### 2.2 風險評估 (Risk Assessment)
| 風險項目 | 潛在影響 | 因應策略 |
| :--- | :--- | :--- |
| 故事分支過多導致邏輯衝突 | 高 | 開發前使用流程圖軟體先規劃好所有邏輯路徑與分支。 |
| 登入系統與資料庫連接不穩 | 中 | 提前進行壓力測試，並使用輕量化、正規化的資料庫結構。 |
| 開發進度趕不上時程 | 高 | 優先完成核心「存檔點」與「登入」等 MVP 功能，後續再擴充故事細節。 |

## 4. 非功能性需求 (Non-Functional Requirements)

- **效能 (Performance)**：故事文字與轉場圖片載入時間應小於 2 秒，確保閱讀流暢不卡頓。
- **安全性 (Security)**：使用者密碼應進行雜湊（Hash）加密處理，確保登入資訊與使用者資料庫安全。
- **易用性 (Usability)**：使用者介面 (UI) 應直覺簡單，符合年輕族群的操作習慣，確保使用者能快速上手。
- **觀賞性 (Aesthetics)**：在不同劇情情境中可切換適合的音樂、提供精美動畫與響應式介面設計。

## 5. 技術架構 (Technical Architecture)

- **前端 (Frontend)**：HTML / CSS / JavaScript（純原生前端，無使用 React/Vue 等框架，符合課堂規範）
- **後端 (Backend)**：Python + Flask（負責 API 路由、資料處理與業務邏輯）
- **資料庫 (Database)**：MySQL（用於儲存使用者帳號、存檔進度、成就解鎖紀錄等關聯性資料）
- **開發工具與版本控制**：GitHub / Antigravity / Gemini
- **團隊溝通管道**：Discord / LINE / Google Meet
